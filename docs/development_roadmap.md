# AIPower-Efficiency-Pilot 开发路线图 (业务逻辑修正版)

## 1. 开发理念 (Business-Driven Design)
本项目开发严格遵循 **“采集感知 -> 动态计费 -> 智能分析 -> 治理闭环”** 的业务逻辑。通过 **“分层存储 (Redis + MySQL + Prometheus)”** 架构，实现财务级的资源效能治理。

## 2. 第一阶段：全维感知与资源池打标 (Perception & Identity)
*   **业务目标：** 解决“谁在哪个池子消耗了多少”的问题。
*   **核心关联文档：** 🏷️ **[感知与打标流程详解](perception_and_tagging_logic.md)**
*   **核心任务：**
    1.  **工程骨架初始化：** 建立 `backend/` (Go) 与 `frontend/` (Next.js) 项目环境。
    2.  **池化采集器 (Pool-Collector)：** 
        *   利用 `client-go` 实现基于 Pod 调度路径的 **“资源池 (Pool) + 切分模式 (Slicing Mode)”** 自动化识别。
        *   对接 Prometheus 实现 5 分钟级的 GPU 实时指标探测（显存、算力、特性利用率）。
    3.  **实时状态缓冲区：** 配置 **Redis** 存储当前的 5 分钟级快照，支撑 Dashboard 实时展示。

## 3. 第二阶段：池化计费与 1d 聚合引擎 (Billing & Consolidation)
*   **业务目标：**
    - 💰 **池子维度**：量化各资源池的真实利用率与日均成本，支撑采购与定价决策。
    - 🏢 **业务维度**：按团队/项目下钻，产出部门级成本分摊报告，解决"钱花到哪里去了"的问题。
    - 🔬 **Pod 维度**：精确追溯每个任务的利用率、显存峰值与实际费用，识别高危浪费任务。
*   **核心关联文档：** 💰 **[计费与聚合流程详解](billing_and_aggregation_logic.md)**
*   **核心任务：**
    1.  **指标持久化与缝合 (Metrics Stitching)：**
        *   在 MySQL `life_trace` 表中恢复 Util Avg/Max, Mem Max 等指标字段（Phase 1 仅保留生命周期）。
        *   开发定时缝合任务：回溯 Pod 生命周期窗口，从 Prometheus 调取真实指标并持久化回填。
    2.  **资源池定价引擎 (Pricing Engine)：** 
        *   在 MySQL 维护 **`pool_pricing`** 表（池子基准价 + 切片模式计费权重）。
        *   编写基于资源池维度的 **“按秒计费”** 算法，实现成本分摊。
    3.  **1d 聚合引擎 (Aggregation Engine)：** 
        *   每天凌晨将采样点聚合为一条 **“周期账单快照 (Billing Snapshot)”** 持久化至 MySQL。
        *   计算每日 P95 利用率及周期消耗总额。

## 4. 第二阶段补全：资源池资产管理 (Resource Asset Management)
*   **业务目标：** 将 K8s 底层的逻辑标签转化为可管理的业务资产。
*   **核心关联文档：** 📦 **[资源池资产管理实现逻辑](resource_pool_management_logic.md)**
*   **核心任务：**
    1.  **资产自动发现 (Asset Discovery)：** 采集器感知识别新 Pool-ID 时，自动在数据库“注册”资产元数据。
    2.  **元数据业务化：** 支持为生硬的 Pool-ID 补充业务别名、应用场景（训练/推理）及责任描述。
    3.  **资产盘点清单：** 提供统一的资源池资产视图，作为 FinOps 治理的物理基础。

## 5. 第三阶段：AI 专家诊断与摘要算法 (LLM Insights) ✅ 已完成
*   **业务目标：** 将海量指标翻译为人类可读的优化建议，核心对象是**任务（Pod/PyTorchJob）**。
*   **核心关联文档：** 🤖 **[LLM Insights 实现逻辑](llm_insights_logic.md)**
*   **核心任务：**
    1.  **数据脱水降维：** 将 7 天的”日快照”压缩为适配 LLM 的 **统计特征摘要**。
    2.  **任务画像识别：** 识别任务类型（训练/推理/开发）、优先级（高/中/低）、硬件依赖（NVLink/FP8/MIG/RDMA）。
    3.  **AI Agent 诊断：** 生成带根因分析的 **《任务治理建议报告》**，包含多种建议选项。
    4.  **治理动作类型：** 支持”降配”、”迁移”、”降配+迁移”三种动作。
    5.  **决策逻辑：** 硬件依赖决定迁移边界，优先级影响治理力度。
    6.  **前端交互：** 实现 AI 专家建议卡片，支持单选建议、批准/拒绝操作。

## 6. 第四阶段：可视化看板与闭环治理 (Governance & ROI) 🟡 进行中
*   **业务目标：** 变现治理价值，实现 ROI 闭环。
*   **核心任务：**
    1.  **治理执行器 (Pilot)：** 根据选择的 AI 建议执行降配、迁移等操作。
    2.  **审批流增强：** 推送企业微信/钉钉审批，支持”预览变更 (Dry-run)”。
    3.  **ROI 闭环看板：** 量化治理前后的成本节省数据，展示治理趋势。

## 7. 第五阶段：V2.0 规划 (Future Roadmap)
*   **多维度成本追踪：** 将计费维度由单纯的 GPU 计算扩展至高性能存储（如 CPFS、对象存储带宽）及数据跨区流转的网络开销。
*   **资源碎片的调度优化：** 从“事后治理”走向“调度干预”，结合调度器感知池化碎片形态。
