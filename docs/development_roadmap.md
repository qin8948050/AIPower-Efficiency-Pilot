# AIPower-Efficiency-Pilot 开发路线图 (业务逻辑修正版)

## 1. 开发理念 (Business-Driven Design)
本项目开发严格遵循 **“采集感知 -> 动态计费 -> 智能分析 -> 治理闭环”** 的业务逻辑。通过 **“分层存储 (Redis + MySQL + Prometheus)”** 架构，实现财务级的资源效能治理。

## 2. 第一阶段：全维感知与资源池打标 (Perception & Identity)
*   **业务目标：** 解决“谁在哪个池子消耗了多少”的问题。
*   **核心关联文档：** 🏷️ **[感知与打标流程详解](perception_and_tagging_logic.md)**
*   **核心任务：**
    1.  **工程骨架初始化：** 建立 `backend/` (Go) 与 `frontend/` (Next.js) 项目环境。
    2.  **池化采集器 (Pool-Collector)：** 
        *   利用 `client-go` 实现基于 Pod 调度路径的 **“资源池 (Pool) + 切分模式 (Slicing Mode)”** 自动化识别。
        *   对接 Prometheus 实现 5 分钟级的 GPU 实时指标探测（显存、算力、特性利用率）。
    3.  **实时状态缓冲区：** 配置 **Redis** 存储当前的 5 分钟级快照，支撑 Dashboard 实时展示。

## 3. 第二阶段：池化计费与 1d 聚合引擎 (Billing & Consolidation)
*   **业务目标：** 将物理消耗变现为财务账单，并实现日级聚合审计。
*   **核心任务：**
    1.  **资源池定价引擎：** 
        *   在 MySQL 维护 **`pool_pricing`** 表（池子基准价 + 切片权重系数）。
        *   编写基于资源池维度的 **“按秒计费”** 算法。
    2.  **1d 聚合引擎：** 
        *   每天凌晨将采样点聚合为一条 **“日级能效/账单快照 (Daily Snapshot)”** 持久化至 MySQL。
        *   计算每日 P95 利用率、Max 显存占用及分摊成本。
    3.  **活跃任务对账：** 建立 `active_sessions` 机制，防止短生命周期 Pod 漏单。

## 4. 第三阶段：AI 专家诊断与摘要算法 (LLM Insights)
*   **业务目标：** 将海量指标翻译为人类可读的优化建议。
*   **核心任务：**
    1.  **数据脱水降维：** 将 7 天的“日快照”压缩为适配 LLM 的 **统计特征摘要**。
    2.  **AI Agent 诊断：** 引入大模型，基于“日聚合摘要”生成带根因分析的 **《跨池迁移与缩盈建议报告》**。
    3.  **硬件特性审计：** 自动识别高端特性（如 TF32/NVLink）的真实利用效能。

## 5. 第四阶段：可视化看板与闭环治理 (Governance & ROI)
*   **业务目标：** 变现治理价值，实现 ROI 闭环。
*   **核心任务：**
    1.  **Next.js Dashboard：** 展示各池子的成本分布、效能红黑榜及 AI 专家卡片。
    2.  **治理执行器 (Pilot)：** 实现基于 AI 建议的“一键调优”功能。引入 **“人机协同审批流 (Human-in-the-loop/Dry-run)”**，在自动通过 K8s API 调整规格或执行跨池迁移前，通过 IM 推送人工确认。
    3.  **管理员调价后台：** 提供图形化的资源池定价管理与策略阈值配置。
    
## 6. 第五阶段：V2.0 规划 (Future Roadmap)
*   **多维度成本追踪：** 将计费维度由单纯的 GPU 计算扩展至高性能存储（如 CPFS、对象存储带宽）及数据跨区流转的网络开销。
*   **资源碎片的调度优化：** 从“事后治理”走向“调度干预”，结合调度器感知池化碎片形态。
